# Hands-On Mini Project: Real-Time Loan Event Streaming Pipeline (Pub/Sub)

## Business Use Case & Purpose

### The Problem

In the batch pipeline (Step 10), we process loans **once daily at 6 AM**. But in reality:

- A fraud team needs to flag suspicious loan applications **within seconds**
- Risk managers want to see a **live dashboard** of incoming applications
- Credit decisions shouldn't wait 24 hours — customers expect **instant responses**
- Regulators require **real-time monitoring** of high-risk loan volumes

**Batch is not enough** when business decisions depend on time-sensitive data.

### What We're Building

A **real-time streaming pipeline** that processes loan events the instant they happen:

```
Loan Application Submitted (Event)
   ↓
Pub/Sub Topic (Message Queue) — receives events in real-time
   ↓
Dataflow Streaming (Apache Beam) — processes each event as it arrives
   ↓
BigQuery (Real-Time Table) — queryable within seconds of event
   ↓
Pub/Sub Alert Topic — triggers notifications for high-risk loans
```

### Batch vs Streaming — When to Use Each

| Aspect | Batch (File 10) | Streaming (This File) |
|--------|----------------|----------------------|
| **Latency** | Hours (runs once/day) | Seconds (processes immediately) |
| **Source** | CSV files in GCS | Events from Pub/Sub |
| **Use case** | Daily reports, historical analysis | Real-time alerts, live dashboards |
| **Cost** | Cheaper (runs briefly) | More expensive (always running) |
| **Complexity** | Simpler | Handles late data, windowing |
| **Example** | "Yesterday's loan summary" | "Alert: 5 high-risk loans in last 10 min" |

### Key Streaming Concepts Demonstrated

1. **Pub/Sub** — Fully managed message queue. Publishers send events, subscribers consume them. Decouples producers from consumers.

2. **Windowing** — Groups unbounded streaming data into finite chunks (e.g., "count loans in the last 5 minutes"). Without windowing, you can't aggregate infinite streams.

3. **Watermarks** — Beam's mechanism to handle late-arriving data. "I believe all events up to time T have arrived."

4. **Exactly-once processing** — Dataflow guarantees each message is processed exactly once (no duplicates, no lost messages).

```sql
-- Real-time loans table — receives data within seconds of application
-- PARTITION BY _PARTITIONTIME = BigQuery auto-partitions by ingestion time
-- This is ideal for streaming since we don't control when data arrives
CREATE TABLE IF NOT EXISTS `project.loan_streaming.realtime_loans` (
    loan_id STRING,
    customer_id STRING,
    loan_type STRING,
    amount FLOAT64,
    interest_rate FLOAT64,
    term_months INT64,
    status STRING,

CLUSTER BY region, risk_category;

-- Windowed aggregation table — updated every 5 minutes with rolling metrics

    window_start TIMESTAMP,
    window_end TIMESTAMP,
    region STRING,
    loan_type STRING,
    application_count INT64,
    total_amount FLOAT64,
    avg_credit_score FLOAT64,
    high_risk_count INT64              -- Count of VERY_HIGH risk in this window
)
PARTITION BY DATE(window_start);
```

## Step 3: Event Publisher (Simulates Real Loan Applications)

```python
# publish_loan_events.py
# Purpose: Simulates a loan application system sending events to Pub/Sub in real-time
# In production, this would be your actual loan application backend/API

import json
import random
import time
from datetime import datetime
from google.cloud import pubsub_v1

PROJECT_ID = "your-gcp-project"
TOPIC_ID = "loan-applications"

# Pub/Sub publisher client — handles batching and delivery
publisher = pubsub_v1.PublisherClient()
topic_path = publisher.topic_path(PROJECT_ID, TOPIC_ID)

LOAN_TYPES = ["PERSONAL", "AUTO", "MORTGAGE", "CREDIT_CARD", "STUDENT"]
REGIONS = ["US_EAST", "US_WEST", "US_CENTRAL", "APAC", "EMEA"]


def generate_loan_event():
    """Create a single fake loan application event (mimics what a real app would send)."""
    return {
        "loan_id": f"LN-{random.randint(1, 9999999):07d}",
        "customer_id": f"CUST-{random.randint(1, 5000):06d}",
        "loan_type": random.choice(LOAN_TYPES),
        "amount": round(random.uniform(1000, 500000), 2),
        "interest_rate": round(random.uniform(3.0, 18.0), 2),
        "term_months": random.choice([12, 24, 36, 48, 60, 120, 240, 360]),
        "status": "PENDING",  # All new applications start as PENDING
        "region": random.choice(REGIONS),
        "credit_score": random.randint(300, 850),
        "event_timestamp": datetime.utcnow().isoformat() + "Z",  # ISO 8601 format
    }


def publish_events(num_events=100, delay_seconds=0.5):
    """
    Publish loan events to Pub/Sub at a controlled rate.
    delay_seconds controls how fast events arrive (simulates real traffic).
    """
    print(f"Publishing {num_events} events to {topic_path}...")
    
    for i in range(num_events):
        event = generate_loan_event()
        
        # Pub/Sub messages must be bytes — serialize dict to JSON string then encode
        message_bytes = json.dumps(event).encode("utf-8")
        
        # publish() returns a future — call .result() to wait for confirmation
        # Attributes are metadata attached to the message (useful for filtering)
        future = publisher.publish(
            topic_path,
            data=message_bytes,
            # Attributes allow subscribers to filter messages without parsing the body
            loan_type=event["loan_type"],
            region=event["region"],
        )
        
        message_id = future.result()  # Blocks until Pub/Sub acknowledges receipt
        
        if (i + 1) % 10 == 0:
            print(f"  Published {i + 1}/{num_events} events (last msg_id: {message_id})")
        
        time.sleep(delay_seconds)  # Simulate real-world event spacing
    
    print(f"Done! Published {num_events} events.")


if __name__ == "__main__":
    # Publish 100 events, one every 0.5 seconds (simulates ~2 applications/sec)
    publish_events(num_events=100, delay_seconds=0.5)
```

## Step 4: Streaming Dataflow Pipeline

```python
# streaming_loan_pipeline.py
# Purpose: Real-time Apache Beam pipeline that reads from Pub/Sub,
#          enriches loan events, writes to BigQuery, and sends alerts for high-risk loans.
# Key difference from batch: This pipeline NEVER stops — it runs continuously.

import argparse
import json
import logging
import apache_beam as beam
from apache_beam.options.pipeline_options import PipelineOptions, StandardOptions
from apache_beam.transforms.window import FixedWindows
from apache_beam.transforms.trigger import AfterWatermark, AfterProcessingTime, AccumulationMode
from datetime import datetime


# --- STEP A: Parse Pub/Sub JSON message into a Python dict ---
class ParsePubSubMessage(beam.DoFn):
    """Deserialize the JSON bytes from Pub/Sub into a Python dictionary."""
    def process(self, message):
        try:
            # message.data is bytes — decode to string, then parse JSON
            record = json.loads(message.decode("utf-8"))
            yield record
        except (json.JSONDecodeError, UnicodeDecodeError) as e:
            # Dead-letter: unparseable messages go to side output
            logging.error(f"Failed to parse message: {e}")
            yield beam.pvalue.TaggedOutput("errors", {
                "raw": message.decode("utf-8", errors="replace"),
                "error": str(e),
            })


# --- STEP B: Enrich each loan event with calculated fields ---
class EnrichLoanEvent(beam.DoFn):
    """Add monthly_payment, risk_category, and processing_timestamp to each event."""
    def process(self, record):
        # Calculate monthly payment (EMI formula)
        r = record["interest_rate"] / 100 / 12  # Monthly interest rate
        n = record["term_months"]                # Number of monthly payments
        p = record["amount"]                     # Loan principal
        
        if r > 0:
            # EMI = P * [r(1+r)^n] / [(1+r)^n - 1]
            monthly_payment = p * (r * (1 + r)**n) / ((1 + r)**n - 1)
        else:
            monthly_payment = p / n
        
        record["monthly_payment"] = round(monthly_payment, 2)
        
        # Risk categorization based on FICO credit score
        score = record["credit_score"]
        if score >= 750:
            record["risk_category"] = "LOW"
        elif score >= 650:
            record["risk_category"] = "MEDIUM"
        elif score >= 550:
            record["risk_category"] = "HIGH"
        else:
            record["risk_category"] = "VERY_HIGH"
        
        # Stamp when our pipeline processed this event
        record["processing_timestamp"] = datetime.utcnow().isoformat() + "Z"
        
        yield record


# --- STEP C: Filter high-risk loans for alerting ---
class FilterHighRisk(beam.DoFn):
    """Route VERY_HIGH risk loans to alert topic. Also flags large loan amounts."""
    def process(self, record):
        # Alert if: very high risk OR extremely large loan amount
        if record["risk_category"] == "VERY_HIGH" or record["amount"] > 400000:
            alert = {
                "alert_type": "HIGH_RISK_APPLICATION",
                "loan_id": record["loan_id"],
                "customer_id": record["customer_id"],
                "amount": record["amount"],
                "credit_score": record["credit_score"],
                "risk_category": record["risk_category"],
                "region": record["region"],
                "timestamp": record["processing_timestamp"],
            }
            yield alert


# --- STEP D: Format windowed aggregations for BigQuery ---
class FormatWindowedMetrics(beam.DoFn):
    """Convert windowed aggregation results into BigQuery-ready dicts."""
    def process(self, element, window=beam.DoFn.WindowParam):
        key, metrics = element  # key = (region, loan_type), metrics = aggregated values
        region, loan_type = key
        
        yield {
            "window_start": window.start.to_utc_datetime().isoformat() + "Z",
            "window_end": window.end.to_utc_datetime().isoformat() + "Z",
            "region": region,
            "loan_type": loan_type,
            "application_count": metrics["count"],
            "total_amount": metrics["total_amount"],
            "avg_credit_score": metrics["avg_credit_score"],
            "high_risk_count": metrics["high_risk_count"],
        }


# --- STEP E: Custom CombineFn for windowed aggregation ---
class LoanMetricsCombiner(beam.CombineFn):
    """
    Accumulates loan metrics within a window.
    CombineFn works like a reducer — processes partial results across workers.
    """
    def create_accumulator(self):
        # Initial state: (count, total_amount, total_credit_score, high_risk_count)
        return (0, 0.0, 0.0, 0)
    
    def add_input(self, accumulator, element):
        # Add one loan record to the running totals
        count, total_amount, total_score, high_risk = accumulator
        count += 1
        total_amount += element["amount"]
        total_score += element["credit_score"]
        if element.get("risk_category") == "VERY_HIGH":
            high_risk += 1
        return (count, total_amount, total_score, high_risk)
    
    def merge_accumulators(self, accumulators):
        # Merge partial results from different workers (parallelism)
        counts, amounts, scores, risks = zip(*accumulators)
        return (sum(counts), sum(amounts), sum(scores), sum(risks))
    
    def extract_output(self, accumulator):
        # Final result for this window
        count, total_amount, total_score, high_risk = accumulator
        return {
            "count": count,
            "total_amount": round(total_amount, 2),
            "avg_credit_score": round(total_score / count, 1) if count > 0 else 0,
            "high_risk_count": high_risk,
        }


# --- MAIN PIPELINE ---
def run(argv=None):
    parser = argparse.ArgumentParser()
    parser.add_argument("--input_subscription", required=True,
                        help="Pub/Sub subscription: projects/PROJECT/subscriptions/SUB")
    parser.add_argument("--output_table", required=True,
                        help="BQ table: project:dataset.table")
    parser.add_argument("--alert_topic", required=True,
                        help="Pub/Sub topic for alerts: projects/PROJECT/topics/TOPIC")
    parser.add_argument("--metrics_table", required=True,
                        help="BQ table for windowed metrics")
    
    known_args, pipeline_args = parser.parse_known_args(argv)
    options = PipelineOptions(pipeline_args)
    
    # CRITICAL: Set streaming=True — this makes the pipeline run continuously
    options.view_as(StandardOptions).streaming = True
    
    # BigQuery schema for the real-time loans table
    realtime_schema = (
        "loan_id:STRING,customer_id:STRING,loan_type:STRING,"
        "amount:FLOAT64,interest_rate:FLOAT64,term_months:INT64,"
        "status:STRING,region:STRING,credit_score:INT64,"
        "risk_category:STRING,monthly_payment:FLOAT64,"
        "event_timestamp:TIMESTAMP,processing_timestamp:TIMESTAMP"
    )
    
    # BigQuery schema for windowed metrics
    metrics_schema = (
        "window_start:TIMESTAMP,window_end:TIMESTAMP,"
        "region:STRING,loan_type:STRING,"
        "application_count:INT64,total_amount:FLOAT64,"
        "avg_credit_score:FLOAT64,high_risk_count:INT64"
    )
    
    with beam.Pipeline(options=options) as p:
        
        # ═══════════════════════════════════════════════════════════════
        # PHASE 1: Read from Pub/Sub (continuous — never ends)
        # ═══════════════════════════════════════════════════════════════
        raw_messages = (
            p
            | "ReadPubSub" >> beam.io.ReadFromPubSub(
                subscription=known_args.input_subscription
            )
        )
        
        # ═══════════════════════════════════════════════════════════════
        # PHASE 2: Parse JSON messages
        # ═══════════════════════════════════════════════════════════════
        parsed = (
            raw_messages
            | "ParseJSON" >> beam.ParDo(ParsePubSubMessage())
                .with_outputs("errors", main="valid")
        )
        
        # ═══════════════════════════════════════════════════════════════
        # PHASE 3: Enrich each loan event
        # ═══════════════════════════════════════════════════════════════
        enriched = (
            parsed["valid"]
            | "Enrich" >> beam.ParDo(EnrichLoanEvent())
        )
        
        # ═══════════════════════════════════════════════════════════════
        # PHASE 4: Write ALL enriched records to BigQuery (real-time)
        # ═══════════════════════════════════════════════════════════════
        enriched | "WriteToBQ" >> beam.io.WriteToBigQuery(
            known_args.output_table,
            schema=realtime_schema,
            # WRITE_APPEND for streaming — continuously adding new rows
            write_disposition=beam.io.BigQueryDisposition.WRITE_APPEND,
            create_disposition=beam.io.BigQueryDisposition.CREATE_IF_NEEDED,
            # Use STREAMING_INSERTS for low-latency writes (data available in seconds)
            # Alternative: FILE_LOADS is cheaper but has higher latency
            method=beam.io.WriteToBigQuery.Method.STREAMING_INSERTS,
        )
        
        # ═══════════════════════════════════════════════════════════════
        # PHASE 5: Send alerts for high-risk loans to Pub/Sub alert topic
        # ═══════════════════════════════════════════════════════════════
        alerts = (
            enriched
            | "FilterHighRisk" >> beam.ParDo(FilterHighRisk())
        )
        
        # Publish each alert as a JSON message to the alert topic
        (
            alerts
            | "SerializeAlert" >> beam.Map(lambda x: json.dumps(x).encode("utf-8"))
            | "PublishAlert" >> beam.io.WriteToPubSub(topic=known_args.alert_topic)
        )
        
        # ═══════════════════════════════════════════════════════════════
        # PHASE 6: Windowed aggregation — metrics every 5 minutes
        # Groups events into 5-minute windows and computes summary stats
        # ═══════════════════════════════════════════════════════════════
        windowed_metrics = (
            enriched
            # Apply 5-minute fixed windows (each window = 300 seconds of data)
            | "Window5Min" >> beam.WindowInto(
                FixedWindows(5 * 60),  # 300 seconds = 5 minutes
                # Trigger: fire when watermark passes window end, OR after 2 min of processing time
                # This ensures we get results even if some events are slightly late
                trigger=AfterWatermark(
                    late=AfterProcessingTime(2 * 60)  # Allow 2 min for late data
                ),
                accumulation_mode=AccumulationMode.DISCARDING,  # Don't re-emit old results
            )
            # Group by (region, loan_type) within each window
            | "KeyByRegionType" >> beam.Map(
                lambda x: ((x["region"], x["loan_type"]), x)
            )
            # Aggregate metrics for each group within the window
            | "Aggregate" >> beam.CombinePerKey(LoanMetricsCombiner())
            # Format for BigQuery
            | "FormatMetrics" >> beam.ParDo(FormatWindowedMetrics())
        )
        
        # Write windowed metrics to BigQuery
        windowed_metrics | "WriteMetrics" >> beam.io.WriteToBigQuery(
            known_args.metrics_table,
            schema=metrics_schema,
            write_disposition=beam.io.BigQueryDisposition.WRITE_APPEND,
            create_disposition=beam.io.BigQueryDisposition.CREATE_IF_NEEDED,
            method=beam.io.WriteToBigQuery.Method.STREAMING_INSERTS,
        )


if __name__ == "__main__":
    logging.getLogger().setLevel(logging.INFO)
    run()
```

## Step 5: Alert Handler (Cloud Function)

```python
# cloud_function/main.py
# Purpose: Triggered by Pub/Sub alert topic — sends notification when high-risk loan detected
# Deployment: gcloud functions deploy loan_alert_handler --trigger-topic=loan-alerts

import base64
import json
import logging
from google.cloud import bigquery

# In production, you'd also import sendgrid/slack SDK for real notifications


def loan_alert_handler(event, context):
    """
    Cloud Function triggered by Pub/Sub message on 'loan-alerts' topic.
    
    Args:
        event: Pub/Sub message (contains 'data' field as base64-encoded JSON)
        context: Event metadata (message_id, timestamp, etc.)
    """
    # Decode the Pub/Sub message (arrives as base64-encoded bytes)
    message_data = base64.b64decode(event["data"]).decode("utf-8")
    alert = json.loads(message_data)
    
    logging.info(f"🚨 HIGH RISK ALERT: Loan {alert['loan_id']}")
    logging.info(f"   Customer: {alert['customer_id']}")
    logging.info(f"   Amount: ${alert['amount']:,.2f}")
    logging.info(f"   Credit Score: {alert['credit_score']}")
    logging.info(f"   Risk: {alert['risk_category']}")
    logging.info(f"   Region: {alert['region']}")
    
    # --- In production, you would: ---
    # 1. Send Slack notification to #fraud-alerts channel
    # 2. Send email to risk management team
    # 3. Write to an alerts audit table in BigQuery
    # 4. Trigger a review workflow in your case management system
    
    # Example: Write alert to BigQuery audit table
    client = bigquery.Client()
    table_id = f"{client.project}.loan_streaming.alerts_audit"
    
    rows_to_insert = [{
        "alert_type": alert["alert_type"],
        "loan_id": alert["loan_id"],
        "customer_id": alert["customer_id"],
        "amount": alert["amount"],
        "credit_score": alert["credit_score"],
        "risk_category": alert["risk_category"],
        "region": alert["region"],
        "alert_timestamp": alert["timestamp"],
    }]
    
    errors = client.insert_rows_json(table_id, rows_to_insert)
    if errors:
        logging.error(f"BigQuery insert errors: {errors}")
    
    return "Alert processed"
```

## Key Interview Talking Points

### "Why Pub/Sub instead of Kafka?"

| Factor | Pub/Sub | Kafka |
|--------|---------|-------|
| Managed | Fully (no servers) | Self-managed or Confluent |
| Scaling | Auto (no partitions to manage) | Manual partition tuning |
| Ordering | Per-key with ordering keys | Per-partition |
| Retention | 7 days default (configurable to 31) | Configurable |
| Cost | Pay per message | Pay for cluster uptime |
| Best for | GCP-native, serverless | Multi-cloud, high-throughput |

### "How do you handle late data?"

> "I use Apache Beam's windowing with AfterWatermark triggers. The watermark tracks event-time progress. Late events (arriving after the window closes) trigger a late-firing that updates the aggregation. I set `accumulation_mode=DISCARDING` so downstream doesn't get duplicate counts."

### "How do you ensure exactly-once processing?"

> "Dataflow provides exactly-once semantics by default for Pub/Sub sources. It uses checkpointing and message deduplication. If a worker crashes mid-processing, the message isn't acknowledged and gets redelivered to another worker — but Beam's runner ensures the output isn't duplicated."

## How This Connects to the Batch Pipeline (File 10)

```
                    ┌─────────────────────────────────┐
                    │     REAL-TIME (This File)        │
                    │  Pub/Sub → Dataflow Streaming    │
                    │  → BigQuery (realtime_loans)     │
                    │  → Alerts (immediate)            │
                    └──────────────┬──────────────────┘
                                   │
                                   │  Same BigQuery warehouse
                                   │
                    ┌──────────────┴──────────────────┐
                    │     BATCH (File 10)              │
                    │  GCS → Dataflow Batch            │
                    │  → BigQuery (curated_loans)      │
                    │  → Daily Summary (scheduled)     │
                    └─────────────────────────────────┘
```

**Lambda Architecture**: Run both pipelines simultaneously:
- Streaming = "speed layer" (immediate but may have small inaccuracies)
- Batch = "batch layer" (daily reconciliation, authoritative numbers)

In practice, many companies run streaming for real-time dashboards/alerts and batch for official reporting/compliance.
